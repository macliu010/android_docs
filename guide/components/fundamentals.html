应用基础

Quickview

  • Android applications are composed of one or more application components (activities, services, content providers, and broadcast receivers)
  • Each component performs a different role in the overall application behavior, and each one can be activated individually (even by other applications)
  • The manifest file must declare all components in the application and should also declare all application requirements, such as the minimum version of Android required and any hardware configurations required
  • Non-code application resources (images, strings, layout files, etc.) should include alternatives for different device configurations (such as different strings for different languages and different layouts for different screen sizes)

Android applications are written in the Java programming language. The Android SDK tools compile the code—along with any data and resource files—into an Android package, an archive file with an .apk suffix. All the code in a single .apk file is considered to be one application and is the file that Android-powered devices use to install the application.

一旦安装在设备上，每一个Android程序会存活在它自己的安全沙箱中:

  • Android操作系统是一个多用户Linux系统，每一个应用都会使用一个不同的用户来运行。
  • 默认情况，系统分配每个应用一个唯一的Linux用户ID(ID只被用在系统中，程序并不知道这个应用ID)。系统给程序中的所有文件设置了 权限，只有分配给程序的用户ID才可以访问到这些文件。
  • 每一个进程有它自己的虚拟机(VM),所以，程序代码运行是独立于其他程序的。
  • 默认，每一个程序在它的Linux进程中运行。当程序的任意组件需要执行Android会启动进程，然后关闭这个进程，当它不再被需要或当系统必须 恢复内存用于其他应用。

在这种情况下，Android系统应用的是最小权限原则。即，每个应用默认只能访问保证其能正常运行所必需的组件。这个原则保证应用不能去访问它未 得到授权的Android系统组件。

尽管如此，还是有一些方法可以让应用与其他应用共享数据，或访问系统服务:

  • 可以安排两个应用共享相同的Linux userID，这样，他们可以互相访问各自的文件。为了保护系统资源，使用相同 User ID的应用同样可以运行在相同的Linux进程中，并且共享相同的VM(应用必须使用相同的证书进行签名)。
  • 应用可以请求权限以访问设备数据，如用户的通讯录、SMS信息、可挂载的存储(SD卡)，相机，蓝牙等。应用的所有权限必须 在安装的时候就由用户授权。

上面介绍了关于一个Android应用如何运行在系统中的一些基础知识。接下来我们会为你介绍：

  • 用以定义Andorid应用的核心框架组件。
  • 用以申明应用的组件及所需设备特征的资源配置文件（manifest）。
  • 用以使应用可更加优雅地适配各种不同设备配置的资源文件，这些资源与应用的代码是隔离的。

For each document in the Application Fundamentals, the Beginner's Path points you to the document we suggest you read next, in order to get up to speed on the core Android concepts.

应用组件

应用组件是一个Andriod应用不可或缺的基础。每个组件都是一个系统可进入你的应用的入口点。虽然并非所有的组件都是一个 用户进入应用的实际入口点，但每个组件都有着特定的作用，都是一个唯一的基础元件，帮助定义应用的整个行为。

总共有四种不同的应用组件。每种都有其特定的功能及特定的生命周期（创建及销毁的时机）。

下面就是这四种应用组件：

活动（Activities）
一个 活动 代表用户界面的一屏。 例如，一个邮件应用可能包含：一个活动/动作用以显示新邮件列表，一个活动/动作用以写新邮件，另外一个活动用以 读取邮件。尽量应用中的这些所有活动一起运行，为用户提供了一个紧凑、完整的用户体验，但活动相互之间是独立的。就此而言，任意其他的应用只要得到该邮件应用 的授权就可以启动这些活动中的任意一个。例如：一个相机应用可以启动邮件应用中的发新邮件活动，以便用户通过邮件分享照片。

活动都是 Activity 的子类实现，你可通过 Activities了解更多帮助内容。

服务（Services）
服务 是运行于后台的组件，用以运行需长期运行的操作或远程进程，且不提供用户界面。例如，一个服务可能用以当用户在其他应用时播放背景音乐，或者用户 在执行其他操作（activity）时从网络获取数据。其他的组件，如活动组件，可以启动一个服务使其运行或绑定一个服务以与其交互。

服务都是 Service 的子类实现，你可通过 Services 了解更多帮助内容。

内容提供程序
内容提供程序 用以管理应用数据的一个共享集合。 你可将数据存储于文件系统，SQLite数据库，web上或其他应用可访问的持久化存储上。通过内容提供 程序，其他应用可以查询甚至修改数据（如果内容提供程序允许）。例如，Android系统提供一个内容提供程序用以管理用户的联系人信息。这样的话，任何有合适权限的应用 都可查询内容提供程序的一部分(如 ContactsContract.Data) 以读取、写入关于特定联系人的信息。

内容提供程序同样可用于读取、写入一些应用私有的、非共享的数据。 例如， 记事本 使用内容提供程序来保存笔记信息。

内容提供程序是 ContentProvider的子类实现， 而且必须实现一套标准API集合以使其他应用可与其交互。更多信息，请查看 内容提供器 开发帮助文档。

广播接收器（Broadcast receivers）
A 广播接收器 是一个用于响应系统范围广播通知的组件。许多广播信息都源自于系统 —— 例如，广播告知屏幕关闭了，电池量低或一张图片被捕获到。 应用也可以发起一个广播 —— 例如，通知其他应用一些数据已下载到设备中了，可供它们使用。尽管广播接收器并不会显示一个用户界面，但它可在广播事件发生时 创建一个通知状态条 来提醒用户。在更多的情况下，广播接收器仅仅是一个其他组件的“网关”，并只被用以做很少的一些工作。例如，它可能用以触发一个基于其通知事件的服务。

广播接收器都是 BroadcastReceiver 的子类实现， 并且每个广播信息都以 Intent 对象进行传送。更多帮助信息请见 BroadcastReceiver 类。

Android系统设计上的一个独特方面在于，任何应用都可以启动其他应用的组件。例如，如果你希望用户利用设备摄像头拍照，可能已经有一个其他应用 可以做到，你的应用只需直接使用该应用的功能即可，而无需额外开发。为此，你无需要将你的代码和拍照应用的代码整合在一起，甚至无需链接到拍照应用， 只需简单地启动照相应用拍照的活动（activity），完成后，照片自动返回给你的应用。从用户角度来看，拍照功能就是你的应用中的一部分。

当系统启动一个组件，它会启动其所属应用的进程（如果进程尚未运行）并实例化改组件运行所需的类。例如，如果你的应用启动相机应用中的拍照活动（activity）， 拍照活动组件将运行在相机应用的进程中，而非你的应用进程中。因此，与大部分其他系统中应用不同，Android应用没有一个入口点（例如，没有main()函数）。

由于Andorid系统中，每个应用都运行在相互独立的进程中，由于文件权限的限制，你的应用不能直接激活其他应用的组件。你只能通过发送一个信息给Android系统， 消息中指明你希望intent启动的组件。Andorid系统会替你激活相应的组件。

激活组件

四种组件类型中的三种：活动（activity）、服务（service）和广播接收器（broadcast receiver）都是通过一个称为intent的异步消息来激活的。 intent在运行时将组件相互绑定（你可将intent看作一个从其他组件请求一个操作的信使），不管这个组件属于你的应用还是其他应用。

intent通过一个Intent对象来进行创建，定义一条用以激活指定组件或 指定类型组件（intent可以是显式的，也可以是隐式的）的消息。

对于活动（activiyt）和服务（service）而言，一条intent消息定义了需要执行的操作（如查看或发送某些事务），也可能指定一个需要操作数据的URI（除了组件启动时 就需要获取的数据之外）。例如，一条intent消息可能为activity传递一个显示一张图片或打开一个web页面的请求。在一些情况下，你可启动一个activity以接受一个结果 ，activity也会以一个Intent对象的方式返回结果。例如，你可以发布一条intent消息让用户选择一个联系人，并将其也通过一条intent消息返回给你 —— 返回的intent消息包含指向所选联系人的URI。

对于广播接收器而言，intent消息只是简单定义需要被广播的通知（例如，一条提示电池量不足的广播仅仅包含一条“battery is low”的字符串）。

其他的组件类型 —— 内容提供器，不是北intents消息所激活，而是在得到一个 ContentResolver 请求后被激活。 content resolver处理所有与内容提供器直接交互的事件。所以，与内容提供器进行交互的其他组件无需intent消息，而是通过调用ContentResolver对象的方法来进行交互。这样也使得内容提供器和请求信息的组件之间有一个抽象层（出于安全的考虑）。

分别有以下几个方法用以激活每种类型的组件：

  • 可通过传输一条 Intent 消息给 startActivity() 或通过 startActivityForResult() (当你希望activity返回一个结果) 来启动一个activity或希望它做一下其他事情。
  • 可通过传输一条 Intent 消息给 startService()来启动一个service（或给运行中的service指定一些新的操作命令）。也可通过传输一条 Intent 消息给 bindService()来绑定一个service。
  • 可通过传输一条 Intent 消息给诸如 sendBroadcast(), sendOrderedBroadcast(), 或 sendStickyBroadcast()这样的方法来初始化一条广播消息。
  • 可通过调用ContentResolver 的 query() 方法向内容提供器执行一条查询。

更多关于intent消息的内容请见 Intents and Intent Filters document. More information about activating specific components is also provided in the following documents: Activities, Services, BroadcastReceiver and content provider.

For more information about using intents, see the Intents and Intent Filters 文档。更多关于激活指定类型组件的内容请见以下文档： Activities, Services, BroadcastReceiver 和 Content Providers。

资源配置文件

在Android系统启动一个应用组件前，系统需要通过读取应用的AndroidManifest.xml文件来了解改应用所包含的组件。 应用必须在这个文件中申明其所有的组件，而且文件也必须位于应用工程目录的根目录下。

除了申明应用的所有组件外，manifest文件还会做以下一些事情：

  • 说明应用所需的所有用户权限，如网络访问权限或读取用户联系人信息的权限。
  • 说明应用所使用API中的最低 API 层次。
  • 说明应用所使用或所需要的软、硬件特性，如摄像头、蓝牙或多点触摸屏。
  • 应用需要链接到的API库（除Android框架API之外），如 Google 地图API库。
  • 其他

申明组件

manifest文件的主要任务是告知系统应用的所有组件。例如，一个manifest文件可以按如下形式申明一个activity： 

<?xml version="1.0" encoding="utf-8"?>
<manifest ... >
    <application android:icon="@drawable/app_icon.png" ... >
        <activity android:name="com.example.project.ExampleActivity"
                  android:label="@string/example_label" ... >
        </activity>
        ...
    </application>
</manifest>

在<application> 元素中， android:icon 属性指向标识应用的图标资源。

在<activity> 元素中， android:name 属性指定 Activity 子类的权限定类名， android:label 属性指定一个用以作为用户可见的activity标签的字符串。

你必须以下面的方式申明所有的应用组件：

  • <activity> 活动的详情
  • <service> 服务的详情
  • <receiver> 广播接收器的详情
  • <provider> 内容提供器的详情

如果你的源码中包含的活动、服务和内容提供器未在manifest中申明，那么它们对系统是不可见的，所以也永远不会执行。但是，广播接收器既可以在manifest文件中定义， 也可以在代码中动态创建（如BroadcastReceiver对象）并调用 registerReceiver()注册到系统中。

更多关于如何为你的应用构建manifest文件的信息请见 The AndroidManifest.xml File。

申明组件功能

如上所述，Activating Components, 你可使用一条 Intent 消息来启动活动、服务和广播接收器，通过在intent消息中显式指定目标组件的方式（使用组件类名）。但是，intent消息更大的作用在于其action的概念。通过intent消息的action，你只需简单地描述你希望执行的操作类型（你也可加上你所希望操作的数据）并允许系统依此在设备中查找得到一个可以执行改action的组件并启动它。如果有多个组件可执行该action，用户可从中选择一个执行。

系统判别组件是否可以影响一个intent消息的方法在于：将接收到的intent消息和manifest文件中申明的intent filters进行比较。

当你在应用的manifest文件中申明一个组件时，可将intent过滤器（intent filters）—— 用以申明改组件的功能包括在内，如此，便可对其他应用发送的intent消息进行响应。你可通过添加一个组件申明元素的子元素<intent-filter> 来申明一个intent消息过滤器。

例如，一个邮件应用中，创建新邮件的活动可能在应用的manifest文件中申明intent过滤器以响应发送intent消息（为了发送邮件）。 这样，应用的其他活动可创建一个包含“发送”操作的intent消息，(ACTION_SEND), 当你通过startActivity()调用这个intent时，系统会匹配到邮件应用的“发送”活动并启动它。

更多关于intent过滤器的内容，请见 Intents和Intent filter 。

声明应用需求

现在市场上有大量装有Android系统的设备，但它们并非都提供一样的特性和功能。为了避免你的应用被安装在缺乏某些应用必需特性的设备上，在manifest文件中明确 声明设备及软件需求很重要。这些声明大部分都是提示性的，系统并不会读取它们，但一些其他服务如Google Play会读取这些信息，以便当人们在他们设备上搜索应用时，可以提供一个筛选。

例如，如果你的应用需要摄像头并且使用Android 2.1提供的API(API Level 7)，那么你就需要在manifest文件中将其定义为必需项。如此，没有摄像头或Android版本低于2.1的设备将不能通过Google Play来安装应用。

当然，你也可以声明你的应用需要使用摄像头，但不是必须的。这种情况下，你的应用必须在运行时检测设备是否包含摄像头，如果没有的话，则需要禁用所有相关的应用特性。

下面是一些在设计开发你的应用时需要注意考虑的设备特性：

屏幕大小及像素密度
为了通过屏幕类型来区分设备，Android为每个设备定义了两个特性：屏幕大小（屏幕的物理尺寸）屏幕像素密度（屏幕像素的物理密度，单位为dpi —— 每英寸的点数）。 为了简化不同类型的屏幕配置描述，Android系统将其可选的组：

屏幕大小包括：小、正常、大和超大。
屏幕像素密度包括：低分辨率、中分辨率、高分辨率和超高分辨率。

默认地，应用会兼容所有的屏幕大小和像素密度，因为Android系统会根据你的UI布局和图片资源做相应的调整。但是，你应该为特定的屏幕尺寸创建特定的布局，并为特定 的像素密度提供特定的图片，使用可切换的布局资源，在manifest文件中通过<supports-screens>元素明确声明你的应用所支持的屏幕尺寸。

更多信息，请见 Supporting Multiple Screens。

输入配置
许多设备提供不同类型的用户输入装置，如硬件硬盘、轨迹球或five-way导航pad。如果你的应用需要一个特定的输入硬件，你则应该在manifest文件中通过 <uses-configuration>元素进行声明。但是，这种情况很少见。
设备特性
在一个Android设备上，有很多硬件和软件特性存在或不存在，如摄像头、光敏器件、蓝牙、特定版本的OpenGL或触屏的精度。你应该通过 <uses-feature>元素声明应用所使用的任何特性，因为任何特性 不会在任何Android设备上都有效。
平台版本
不同的Android设备通常运行着不同的Android操作系统版本，如Android 1.6或Android 2.3。 每个后续版本通常包含之前版本不可用的API。为了标识 哪个API集是可用的，每个平台版本指定一个 API Level（如，Android 1.0 是API层次1，Android 2.3则是API层次9）。 如果你使用任何在1.0版本后增加到平台中的API，那么你应该通过<uses-sdk>元素声明最小API级别。

声明你的应用的所有这些需求非常重要，因为，当你将应用发布到Google Play中，Google Play将使用这些声明作为应用是否在某台设备可用的依据。 就此而言，你的应用只会对满足其所有需求的设备可用。

更多关于Google Play如何基于这些需求来过滤应用的信息，请见Filters on Google Play。

应用资源

一个Android应用不仅仅由代码构成，也需要一些和源码无关的资源，如图片，音频文件以及任何与应用可视化展现相关的资源。 例如，你应该通过XML文件定义动画、菜单、样式、颜色和活动的用户界面布局。通过提供可选资源的集合 —— 可优化你的应用在不同设备下（如不同的屏幕尺寸和语言设置）的表现，并使得更新应用的各种特性变得简单，且不用修改代码。

For every resource that you include in your Android project, the SDK build tools define a unique integer ID, which you can use to reference the resource from your application code or from other resources defined in XML. For example, if your application contains an image file named logo.png (saved in the res/drawable/ directory), the SDK tools generate a resource ID named R.drawable.logo, which you can use to reference the image and insert it in your user interface.

对于你的Android项目所包含的每个资源，Android SDK编译工具会定义一个唯一的整数ID。通过这个ID，你可从你的应用代码或其他XML定义的资源中引用改ID定义应的资源。

One of the most important aspects of providing resources separate from your source code is the ability for you to provide alternative resources for different device configurations. For example, by defining UI strings in XML, you can translate the strings into other languages and save those strings in separate files. Then, based on a language qualifier that you append to the resource directory's name (such as res/values-fr/ for French string values) and the user's language setting, the Android system applies the appropriate language strings to your UI.

Android supports many different qualifiers for your alternative resources. The qualifier is a short string that you include in the name of your resource directories in order to define the device configuration for which those resources should be used. As another example, you should often create different layouts for your activities, depending on the device's screen orientation and size. For example, when the device screen is in portrait orientation (tall), you might want a layout with buttons to be vertical, but when the screen is in landscape orientation (wide), the buttons should be aligned horizontally. To change the layout depending on the orientation, you can define two different layouts and apply the appropriate qualifier to each layout's directory name. Then, the system automatically applies the appropriate layout depending on the current device orientation.

For more about the different kinds of resources you can include in your application and how to create alternative resources for various device configurations, see the Application Resources developer guide.
